## 1. What this release delivers

Two concerns, deliberately bundled because they must be designed together:

1. **Backend abstraction** — introduce a narrow `LlmBackend` interface so every tool calls the backend through one neutral contract. Migrate the existing five tools (`summarize`, `summarize-long`, `classify`, `extract`, `transform`) to the abstraction. The current `OllamaClient` survives as the `OllamaBackend` implementation; user-visible behavior is unchanged.

2. **`summarize-long-chunked` tool** — a new MCP tool that splits inputs larger than a single Ollama call can handle, summarizes each chunk, then recursively combines the chunk summaries into a final output. Each individual LLM call stays under ~50 s so it survives Claude Code's ~60 s hard MCP request timeout.

These bundle because (a) the chunking feature is the first consumer of the abstraction and (b) designing the abstraction without a concrete non-trivial consumer risks the "ivory-tower interface" failure mode.

## 2. Out of scope for v0.2.0

Explicitly not in this release:

- **llama.cpp backend implementation.** The abstraction *enables* it; actually adding `LlamaCppBackend` is v0.3.0. No llama.cpp code lands in this release.
- **Behavior change to existing `summarize-long`.** The existing tool keeps its "single LLM call" semantics. Chunking is a new tool, not a silent upgrade.
- **Ollama removal.** Ollama remains the only shipped backend in v0.2.0. Deprecation starts in v0.3.0+ once a second backend exists.
- **Streaming output on the new tool.** Map-reduce doesn't stream naturally; deferred.
- **Cross-language chunk boundary optimization** (e.g., Chinese-aware splitters). Use the framework's recursive-character defaults; re-evaluate if summary quality is visibly poor.
- **Source-URI URLs** for the new tool. `file://` only in v0.2.0 — the HTTP(S) fetch path already has its own size limit that interacts awkwardly with chunking and deserves its own thinking.

## 3. Backend abstraction

### 3.1 Interface contract

```ts
// src/llm/backend.ts — new file

/** Neutral JSON-Schema-like object for grammar-constrained output. */
export type JsonSchemaLike = Record<string, unknown>;

export interface ChatOptions {
  /** Optional system prompt. */
  system?: string;
  /** User prompt (always required). */
  user: string;
  /** Sampling temperature. 0 = deterministic. */
  temperature?: number;
  /** Max tokens the backend should admit from the prompt (Ollama: num_ctx, llama.cpp: -c). */
  maxInputTokens: number;
  /** Max tokens to generate. Absent = backend default. */
  maxOutputTokens?: number;
  /** Grammar-constrained output schema. Interface takes JSON Schema; backend adapts to native format. */
  format?: JsonSchemaLike;
  /** Optional stop sequences. */
  stopSequences?: string[];
}

export interface ChatResult {
  text: string;
  promptTokens: number;
  completionTokens: number;
}

export interface LlmBackend {
  /** Identifier (e.g. "ollama:qwen2.5:7b") — surfaced in telemetry. */
  readonly modelId: string;
  /** Run a chat-style completion. */
  chat(opts: ChatOptions): Promise<ChatResult>;
  /** Lightweight liveness check. Backends that don't need this can resolve immediately. */
  ping(): Promise<void>;
}
```

**What is NOT on the interface:** `keepAlive`, `think`, `numCtx` (renamed to `maxInputTokens`), any Ollama-specific field. Backend-specific concerns live in the concrete class constructor.

**Format handling note:** v0.2.0 passes JSON Schema through unchanged (Ollama's `format:` accepts it). When `LlamaCppBackend` lands, it will translate JSON Schema → GBNF grammar internally. Interface stays stable.

### 3.2 OllamaBackend

Wraps the existing `OllamaClient`. Migration is mechanical:

```ts
// src/llm/ollama-backend.ts — new file

export class OllamaBackend implements LlmBackend {
  constructor(
    private readonly client: OllamaClient,
    private readonly modelTag: string,
    /** Ollama-specific lifecycle hints, kept out of the neutral interface. */
    private readonly keepAlive?: string | number,
  ) {}

  get modelId() { return `ollama:${this.modelTag}`; }

  async chat(opts: ChatOptions): Promise<ChatResult> {
    return this.client.chat({
      model: this.modelTag,
      keepAlive: this.keepAlive,
      numCtx: opts.maxInputTokens,
      numPredict: opts.maxOutputTokens,
      system: opts.system,
      user: opts.user,
      temperature: opts.temperature,
      format: opts.format,
      stopSequences: opts.stopSequences,
    });
  }

  ping() { return this.client.ping(); }
}
```

### 3.3 Per-tool tier → backend resolution

Existing `tierForTool` + `DEFAULT_CONFIG.tiers[tier]` continues to resolve **which model** per tool. A thin factory turns the resolved tier config into a backend instance at tool-invocation time:

```ts
// src/mcp/backend-factory.ts — new file

export function backendForTool(
  client: OllamaClient,
  config: BridgeConfig,
  toolName: string,
): LlmBackend {
  const tier = tierForTool(config, toolName);
  const tcfg = config.tiers[tier];
  return new OllamaBackend(client, tcfg.model, tcfg.keepAlive);
}
```

## 5. `summarize-long-chunked` tool

### 5.1 Tool contract

```
summarize-long-chunked(
  text?:         string,
  source_uri?:   string,         // file:// only in v0.2.0
  style?:        string,
  max_chunks?:   number = 30,    // safety cap; job errors out above this
) → final summary string (+ telemetry footer)
```

- `text` and `source_uri` mutually exclusive, consistent with existing tools.
- `_meta` carries the usual fields plus `chunks_processed`, `map_total_prompt_tokens`, `map_total_completion_tokens`, `reduce_depth` (number of reduce passes).
- Footer appends `chunks=<N>` to the existing format.
- Single-chunk inputs (fit in one Ollama call) skip the reduce phase — map step alone returns final output. This makes the tool a safe superset of `summarize-long`.

### 5.2 Algorithm

```
1. Load source (text | source_uri).
2. chunker.split(text) → [chunk₁, …, chunkₙ]
   - RecursiveCharacterTextSplitter from @langchain/textsplitters
   - Target chunk size: 2000 tokens (empirical sweet spot per cognitivetech)
   - Overlap: 200 tokens (10% — standard)
   - Token counter: js-tiktoken cl100k (OpenAI BPE as proxy; ±15% drift on Qwen)
3. If n == 1: run MAP_PROMPT on the single chunk; return.
4. Else:
   a. MAP phase:
      - p-limit(concurrency=2) wrapping each chunk with MAP_PROMPT
      - Each call: maxInputTokens = 2500 (chunk + prompt + margin), maxOutputTokens = 400
      - Per-call timeout 50 s (hard). Timeouts degrade gracefully — see §7.
   b. REDUCE phase:
      - Join chunk summaries with markers [Chunk 1], [Chunk 2], …
      - If joined length fits REDUCE_PROMPT + margin within maxInputTokens (16K Tier C): one reduce call → done.
      - Else: recurse — chunk the joined summaries and run another MAP+REDUCE pass. Max recursion depth: 3.
```

### 5.3 Concurrency and timeout budget

- **Map concurrency:** `p-limit(2)`. Ollama serializes on one Metal context so higher values add queueing overhead without throughput. Configurable via `OMCP_CHUNK_CONCURRENCY` for future hardware / llama.cpp.
- **Per-call soft timeout:** 50 s. `Promise.race` between the backend call and a 50 s timer. On timeout the chunk gets a placeholder summary (§7); the job continues.
- **Total job soft timeout:** none. The MCP client's own timeout will kill runaway jobs; the tool doesn't impose an artificial ceiling.

### 5.4 Prompts (first drafts)

**MAP_SYSTEM** (per-chunk summarizer):

```
You are summarizing a fragment of a longer document. Produce a concise summary
of just this fragment — NOT the whole document. Preserve concrete facts,
specific numbers, named entities, and any explicit claims. Do not speculate
beyond the fragment. Output 3–5 short sentences in the source language.
```

**MAP_USER** (template):

```
Fragment {i} of {n} from: {title_or_uri}

<fragment>
{chunk_text}
</fragment>

Summary of this fragment:
```

**REDUCE_SYSTEM** (combiner):

```
You are given an ordered list of fragment summaries from one document. Produce
a single coherent summary of the whole document. Requirements: (1) do not
mirror the fragment-list structure — collapse related points into themes;
(2) deduplicate — if two fragments say the same thing, say it once;
(3) preserve specific facts, numbers, and named entities from the fragments;
(4) 1–2 sentence lead + 3–6 themed bullets; (5) source language.
```

**REDUCE_USER** (template):

```
Document: {title_or_uri}

{joined_chunk_summaries}

Final summary:
```

## 7. Failure modes and mitigations

| Failure | Mitigation |
|---|---|
| One chunk's Ollama call times out (> 50 s) | Substitute `[Chunk N: summary unavailable — call timed out]` into the map output. Continue the job. Mark in `_meta.chunks_failed`. |
| One chunk's Ollama call errors (non-timeout) | Same as timeout. Error message captured to `_meta.chunk_errors[N]`. |
| Too many chunks (> max_chunks) | Hard error: `isError: true`, message "input exceeds max_chunks={N}; split the source or raise the limit". No partial work performed. |
| Reduce step's joined summaries exceed its context | Recurse: re-chunk the joined summaries, run another map+reduce pass. Max depth 3. Beyond depth 3: return partial ("Warning: input too large to fully reduce; showing last-pass output."). |
| MCP client disconnects mid-job | Best-effort — the Ollama calls complete in the background; their results are discarded on the bridge side. No cleanup hazard because each call is independent. |
| Source URI read fails | Same as existing tools — `isError: true` with the reader's message. No chunking attempted. |
| `@langchain/textsplitters` breaks change during install | Pinned in package.json. Smoke test T9 is the early-warning signal. |
| js-tiktoken undercounts Qwen tokens by > 20 % | Add a safety factor: target chunk size × 0.85 when measured in js-tiktoken. Revisit once we run `diag-long-input.mjs` style measurements with the new tool. |

## 8. Testing strategy

**Unit tests (vitest, no Ollama required — runs in CI):**

**Integration tests (require Ollama, excluded from CI):**

## 9. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Reduce prompt still produces mirror-the-chunks output | Medium | Medium (summary quality degrades to a list of chunk summaries) | REDUCE_SYSTEM explicitly instructs "do not mirror" — a lesson already learned in v0.1.2 F4 regression fix. Add T9b assertion: output bullet count ≤ 6 even with 30 chunks. |
| Concurrency=2 is the wrong ceiling on some hardware | Low | Low (perf only) | Env-var override. Not a correctness risk. |
| Backend abstraction leaks Ollama assumptions (e.g., a later tool needs `keepAlive`) | Low-Medium | Medium (might force interface change) | Migration of five existing tools is the stress test. If any tool genuinely needs an escape hatch, the interface gets a `backendHint?: Record<string,unknown>` passthrough — NOT resolved now, but reserved. |
| Adding `p-limit` dep surfaces a transitive license issue | Very low | Very low | p-limit is MIT, zero-dep, audited thousands of times. |
| Migration breaks one of the five existing tools subtly | Medium | High (user-visible regression) | `smoke-bridge.mjs` must pass **before** merge. Plus: the migration commit is structured as five separate commits (one per tool), bisectable. |
| js-tiktoken token count drifts from Qwen tokenizer badly enough to bust num_ctx | Low | Medium (some chunks truncated by Ollama) | 15 % safety margin on chunk size. Measurable via `prompt_eval_count` in `_meta` — if we see chunks hitting num_ctx exactly in telemetry, tighten the margin. |
| llama.cpp GBNF grammar will reject some schemas we currently pass to Ollama | Not relevant to v0.2.0 | — | Explicit scope: this is a v0.3.0 concern. Documented here to not be surprised later. |

## 12. Open decisions for Auditor

Points I believe are settled but want explicit sign-off on:

1. **Tool name.** `summarize-long-chunked` (descriptive, ugly) vs `summarize-chunked` (shorter) vs `summarize-long` v2 (overloads existing). My vote: `summarize-long-chunked` — clearest contract, no silent behavior change.
2. **Chunk size default 2000 tokens.** Empirical per cognitivetech. Alternative: 4000 (fewer calls, higher per-call latency). My vote: 2000, with env-var override.
3. **Migration of existing five tools in the same release.** Alternative: migrate only `summarize-long` and defer others. My vote: all five together — the interface is only proven when it's carrying all the load, and splitting the migration creates two inconsistent worlds in the codebase.
4. **Max recursion depth 3.** Above this, return partial. Alternative: unlimited. My vote: 3 — if a document needs deeper reduction it's an outlier that deserves a dedicated tool or manual pre-chunking.
5. **No streaming.** Deferred to a later release. OK to leave out of v0.2.0.
