**Status:** Draft, pending Auditor pass. No code yet.
**Author:** Claude Opus 4.7
**Date:** 2026-04-24

## 1. What this release delivers

Two concerns, deliberately bundled because they must be designed together:

1. **Backend abstraction** — introduce a narrow `LlmBackend` interface so every tool calls the backend through one neutral contract. Migrate the existing five tools (`summarize`, `summarize-long`, `classify`, `extract`, `transform`) to the abstraction. The current `OllamaClient` survives as the `OllamaBackend` implementation; user-visible behavior is unchanged.

2. **`summarize-long-chunked` tool** — a new MCP tool that splits inputs larger than a single Ollama call can handle, summarizes each chunk, then recursively combines the chunk summaries into a final output. Each individual LLM call stays under ~50 s so it survives Claude Code's ~60 s hard MCP request timeout.

These bundle because (a) the chunking feature is the first consumer of the abstraction and (b) designing the abstraction without a concrete non-trivial consumer risks the "ivory-tower interface" failure mode.

## 2. Out of scope for v0.2.0

Explicitly not in this release:

- **llama.cpp backend implementation.** The abstraction *enables* it; actually adding `LlamaCppBackend` is v0.3.0. No llama.cpp code lands in this release.
- **Behavior change to existing `summarize-long`.** The existing tool keeps its "single LLM call" semantics. Chunking is a new tool, not a silent upgrade.
- **Ollama removal.** Ollama remains the only shipped backend in v0.2.0. Deprecation starts in v0.3.0+ once a second backend exists.
- **Streaming output on the new tool.** Map-reduce doesn't stream naturally; deferred.
- **Cross-language chunk boundary optimization** (e.g., Chinese-aware splitters). Use the framework's recursive-character defaults; re-evaluate if summary quality is visibly poor.
- **Source-URI URLs** for the new tool. `file://` only in v0.2.0 — the HTTP(S) fetch path already has its own size limit that interacts awkwardly with chunking and deserves its own thinking.

## 3. Backend abstraction

### 3.1 Interface contract

```ts
// src/llm/backend.ts — new file

/** Neutral JSON-Schema-like object for grammar-constrained output. */
export type JsonSchemaLike = Record<string, unknown>;

export interface ChatOptions {
  /** Optional system prompt. */
  system?: string;
  /** User prompt (always required). */
  user: string;
  /** Sampling temperature. 0 = deterministic. */
  temperature?: number;
  /** Max tokens the backend should admit from the prompt (Ollama: num_ctx, llama.cpp: -c). */
  maxInputTokens: number;
  /** Max tokens to generate. Absent = backend default. */
  maxOutputTokens?: number;
  /** Grammar-constrained output schema. Interface takes JSON Schema; backend adapts to native format. */
  format?: JsonSchemaLike;
  /** Optional stop sequences. */
  stopSequences?: string[];
}

export interface ChatResult {
  text: string;
  promptTokens: number;
  completionTokens: number;
}

export interface LlmBackend {
  /** Identifier (e.g. "ollama:qwen2.5:7b") — surfaced in telemetry. */
  readonly modelId: string;
  /**
   * Run a chat-style completion. Implementations MUST honor `signal.aborted`
   * and abort any in-flight work when the signal fires — long-running chunked
   * jobs rely on this to clean up when the MCP client disconnects.
   */
  chat(opts: ChatOptions, signal?: AbortSignal): Promise<ChatResult>;
  /**
   * Count tokens in `text` according to *this backend's* tokenizer. The
   * chunker uses this to size chunks for the actual model in use. Implementations
   * MAY return an approximate count; the docstring on each implementation MUST
   * declare whether the result is exact or proxy. The chunker assumes ±15 %
   * drift on proxy implementations and applies a safety margin.
   */
  countTokens(text: string): Promise<number>;
  /** Lightweight liveness check. Backends that don't need this can resolve immediately. */
  ping(): Promise<void>;
}
```

**What is NOT on the interface:** `keepAlive`, `think`, `numCtx` (renamed to `maxInputTokens`), any Ollama-specific field. Backend-specific concerns live in the concrete class constructor.

**Format handling note:** v0.2.0 passes JSON Schema through unchanged (Ollama's `format:` accepts it). When `LlamaCppBackend` lands, it will translate JSON Schema → GBNF grammar — a non-trivial translation with no off-the-shelf library. This is an explicit v0.3.0 risk item, flagged here so v0.3.0 planning doesn't treat the translator as a one-day task.

**`maxInputTokens` semantics:** this is a *per-call* budget hint, not a model-load-time parameter. For Ollama it maps directly to `num_ctx` (Ollama pre-allocates KV for the value). For llama-server, the model is loaded with `-c <max>` once; per-call `maxInputTokens` must be ≤ that load-time value. `LlamaCppBackend` will enforce this by either picking a load-time `-c` = 32 K conservatively, or running multiple slots with different `-c`. Interface documentation will state this contract explicitly.

### 3.2 OllamaBackend

Wraps the existing `OllamaClient`. Migration is mechanical:

```ts
// src/llm/ollama-backend.ts — new file

export class OllamaBackend implements LlmBackend {
  constructor(
    private readonly client: OllamaClient,
    private readonly modelTag: string,
    /** Ollama-specific lifecycle hints, kept out of the neutral interface. */
    private readonly keepAlive?: string | number,
  ) {}

  get modelId() { return `ollama:${this.modelTag}`; }

  async chat(opts: ChatOptions, signal?: AbortSignal): Promise<ChatResult> {
    return this.client.chat({
      model: this.modelTag,
      keepAlive: this.keepAlive,
      numCtx: opts.maxInputTokens,
      numPredict: opts.maxOutputTokens,
      system: opts.system,
      user: opts.user,
      temperature: opts.temperature,
      format: opts.format,
      stopSequences: opts.stopSequences,
      signal,  // propagated to the ollama npm client's fetch via AbortController
    });
  }

  /**
   * Approximate — uses js-tiktoken (cl100k) as a proxy. ±15 % drift on Qwen /
   * Llama tokenizers. Yields to the event loop between segments to avoid
   * blocking MCP keep-alive responses on large inputs.
   */
  async countTokens(text: string): Promise<number> {
    // js-tiktoken's encode() is purely synchronous; we cannot interrupt it
    // mid-call. Strategy: slice the raw string into ~20 KB segments, encode
    // each segment, sum the lengths, and yield via setImmediate between
    // segments. Boundary effects (a multibyte character or a multi-character
    // token straddling two slices) introduce ≤ ±2 % miscount on top of the
    // ±15 % tokenizer-proxy drift — well within the safety margins the chunker
    // already applies.
    const SEG = 20_000; // bytes per segment
    let total = 0;
    for (let i = 0; i < text.length; i += SEG) {
      total += encoder.encode(text.slice(i, i + SEG)).length;
      if (i + SEG < text.length) await new Promise(r => setImmediate(r));
    }
    return total;
  }

  ping() { return this.client.ping(); }
}
```

`OllamaClient.chat()` accepts an `AbortSignal` today via the underlying `ollama` npm client's `fetch` implementation — a small internal wiring change rather than a new capability.

**Why `countTokens` lives on the backend, not the chunker.** v0.3.0's `LlamaCppBackend` will use llama-server's exact `/tokenize` endpoint, which is per-model accurate. Hardcoding `js-tiktoken` inside the chunker would force a refactor at v0.3.0; placing it on the backend interface keeps the chunker stable across backend swaps.

**Event-loop yielding — concrete strategy.** Synchronous tokenization of a 50–100 K-token document blocks the Node event loop for seconds — long enough for the MCP transport to miss keep-alive responses and drop the connection before the LLM is even called. `js-tiktoken.encode()` itself is non-interruptible, so the yielding pattern slices the raw string into ~20 KB segments, encodes each one, and yields via `setImmediate` between segments (see code above). Each 20 KB segment is ~5–7 K tokens of work — small enough to keep individual blocking spans under ~10 ms on the target hardware. The boundary-effect miscount (a token straddling two slices) is bounded at ±2 %, on top of the proxy tokenizer's ±15 % drift; the chunker's existing `× 0.85` safety margin absorbs both. The fast-path size check (§5.2 step 3) avoids `countTokens` entirely when a cheap byte-count estimate already proves the input does not fit.

### 3.3 Per-tool tier → backend resolution

Existing `tierForTool` + `DEFAULT_CONFIG.tiers[tier]` continues to resolve **which model** per tool. A thin factory turns the resolved tier config into a backend instance at tool-invocation time:

```ts
// src/mcp/backend-factory.ts — new file

export function backendForTool(
  client: OllamaClient,
  config: BridgeConfig,
  toolName: string,
): LlmBackend {
  const tier = tierForTool(config, toolName);
  const tcfg = config.tiers[tier];
  return new OllamaBackend(client, tcfg.model, tcfg.keepAlive);
}
```

## 5. `summarize-long-chunked` tool

### 5.1 Tool contract

```
summarize-long-chunked(
  text?:         string,
  source_uri?:   string,         // file:// only in v0.2.0
  style?:        string,
  max_chunks?:   number = 100,   // safety cap; job errors out above this
) → final summary string (+ telemetry footer)
```

- `text` and `source_uri` mutually exclusive, consistent with existing tools.
- `_meta` carries the usual fields plus `chunks_processed`, `map_total_prompt_tokens`, `map_total_completion_tokens`, `reduce_depth` (number of reduce passes; 0 when the fast-path single call is used).
- Footer appends `chunks=<N>` to the existing format.
- **Fast-path:** if the full source fits in Tier C's `maxInputTokens` with prompt overhead margin, the tool performs a single call with the normal `summarize-long` prompt and returns — no chunking, no reduce. This makes the tool a strict superset of `summarize-long` without paying the chunking tax on short inputs.
- **Configuration:** chunk size and overlap are tunable via env vars `OMCP_CHUNK_SIZE` (default 2000) and `OMCP_CHUNK_OVERLAP` (default 200). `OMCP_CHUNK_CONCURRENCY` (default 2) caps in-flight map calls.
- **Rationale for `max_chunks = 100`:** a 55 K-token target input at 2000-token chunks produces ~31 chunks; at stricter settings (e.g. `OMCP_CHUNK_SIZE=1000`) it produces ~62. The cap is a runaway-cost guard, not a normal-path limit — it should not be tripped by realistic inputs.

### 5.2 Algorithm

```
0. jobSignal = extra.signal  // from MCP SDK's RequestHandlerExtra.
   The @modelcontextprotocol/sdk v1.29 server already wires client cancellations
   and transport disconnects into this signal — verified at
   node_modules/@modelcontextprotocol/sdk/dist/cjs/shared/protocol.d.ts:177:
       "An abort signal used to communicate if the request was cancelled
        from the sender's side."
   No manual transport-disconnect wiring needed.
1. Load source (text | source_uri).
2. Validate env: OMCP_CHUNK_OVERLAP < OMCP_CHUNK_SIZE; both > 0; concurrency ≥ 1.
   On invalid: hard error before any work.
3. Fast-path size estimate (cheap, NO tokenization):
   estimatedTokens = Math.ceil(byteLength / 3)  // conservative for Chinese; ~bytes/4 for English
   If estimatedTokens ≤ Tier_C.maxInputTokens − overhead − margin:
     // very likely fits; verify with one tokenize call (already-loaded path)
     actualTokens = await backend.countTokens(source)
     If actualTokens + overhead ≤ Tier_C.maxInputTokens:
       run summarize-long's existing single-call path with jobSignal; return.
   Else: proceed to chunking. (We never tokenize the full source if the estimate
   already exceeds the budget — chunking will tokenize per-chunk anyway.)
4. chunker.split(source) → [chunk₁, …, chunkₙ]
   - RecursiveCharacterTextSplitter from @langchain/textsplitters
   - Target chunk size: OMCP_CHUNK_SIZE (default 2000 tokens, cognitivetech empirical sweet spot)
   - Overlap: OMCP_CHUNK_OVERLAP (default 200 tokens)
   - Token counter: backend.countTokens (per §3.1 — for OllamaBackend that's
     js-tiktoken with event-loop yielding; for v0.3.0 LlamaCppBackend that's
     llama-server's /tokenize)
   - If n > max_chunks: error with explicit message including suggested OMCP_CHUNK_SIZE.
5. MAP phase:
   - p-limit(OMCP_CHUNK_CONCURRENCY, default 2) wrapping each chunk function.
   - Each chunk function FIRST checks `if (jobSignal.aborted) throw signal.reason;`
     before any backend call. p-limit does not cancel queued closures on its
     own; without this guard a job-level abort would still drain the queue
     doing useless work.
   - Per-call signal: callSignal = AbortSignal.any([jobSignal, AbortSignal.timeout(50_000)])
     This chains a per-call 50 s soft timeout to the job signal. A per-call
     timeout aborts ONLY that call; the job's signal remains untouched and
     other chunks continue.
   - Each call: maxInputTokens = OMCP_CHUNK_SIZE + 500 (prompt + margin),
     maxOutputTokens = 400
   - Catch logic distinguishes:
       a. jobSignal.aborted → re-throw; whole job exits cleanly
       b. local 50 s timeout (callSignal aborted but jobSignal not) → substitute
          `[Chunk N: summary unavailable — call timed out]` and continue;
          record in _meta.chunks_failed
       c. Backend error (non-abort) → same as timeout case for substitution
6. REDUCE phase:
   - Join chunk summaries with markers [Chunk 1], …, [Chunk n].
   - Per-pass bucket budget: 3 K tokens (REDUCE_BUCKET_TOKENS), NOT 4 K or 8 K.
     Rationale below.
   - If joined reduce input ≤ 3 K tokens: ONE reduce call → done.
   - Else: split chunk summaries into buckets of ≤ 3 K tokens each; reduce
     each bucket (same MAP-style p-limit fan-out, same per-call signal chain);
     recurse with the bucket results.
   - Max recursion depth 3. Beyond: return depth-3 output with `partial: true`
     in _meta.
```

**Why the bucket budget shrinks across drafts: 8 K → 4 K → 3 K.** The first two attempts under-counted generation time. Math at the 50 s soft timeout, on qwen2.5:7b Q4_K_M, with measured ~6.4 s/1 K prompt-eval and ~25 tok/s generation under nominal load (dropping to ~15 tok/s under thermal throttling, common during sustained map-reduce on a 16 GB Mac):

| Bucket | Prompt-eval | Gen (nominal, 25 t/s) | Gen (throttled, 15 t/s) | Total nominal | Total throttled | Verdict |
|---|---|---|---|---|---|---|
| 8 K | 51.2 s | 16 s | 26.7 s | 67.2 s | 77.9 s | ❌ both fail |
| 4 K | 25.6 s | 16 s | 26.7 s | 41.6 s | 52.3 s | ⚠️ throttled fails |
| **3 K** | 19.2 s | 16 s | 26.7 s | 35.2 s | 45.9 s | ✅ ~4 s margin even throttled |

A 400-token reduce output is realistic for a "1–2 sentence lead + 3–6 themed bullets" REDUCE prompt — not worst case. The 3 K budget keeps us safely under 50 s even when the user's machine is under load.

**Reduce depth at target sizes, with 3 K bucket:** 55 K input → 31 chunks × 400 = 12.4 K reduce input → 5 buckets pass 1 → 2 K → 1 call pass 2. `reduce_depth = 2`. 200 K input → 112 chunks × 400 = 44.8 K → 15 buckets pass 1 → 6 K → 2 buckets pass 2 → 0.8 K → 1 call pass 3. `reduce_depth = 3`, exactly at the cap. Above ~200 K tokens we return `partial: true`. Same coverage envelope as the 4 K design, just with safer per-call margins.

### 5.3 Concurrency and timeout budget

- **Map concurrency:** `p-limit(2)`. Ollama serializes on one Metal context so higher values add queueing overhead without throughput. Configurable via `OMCP_CHUNK_CONCURRENCY` for future hardware / llama.cpp.
- **Per-call soft timeout:** 50 s. `Promise.race` between the backend call and a 50 s timer. On timeout the chunk gets a placeholder summary (§7); the job continues.
- **Total job soft timeout:** none. The MCP client's own timeout will kill runaway jobs; the tool doesn't impose an artificial ceiling.

## 7. Failure modes and mitigations

| Failure | Mitigation |
|---|---|
| One chunk's Ollama call times out (> 50 s) | Substitute `[Chunk N: summary unavailable — call timed out]` into the map output. Continue the job. Mark in `_meta.chunks_failed`. |
| One chunk's Ollama call errors (non-timeout) | Same as timeout. Error message captured to `_meta.chunk_errors[N]`. |
| Too many chunks (> max_chunks) | Hard error: `isError: true`, message "input exceeds max_chunks={N}; split the source or raise the limit". No partial work performed. |
| Reduce step's joined summaries exceed its context | Recurse: re-chunk the joined summaries, run another map+reduce pass. Max depth 3. Beyond depth 3: return partial ("Warning: input too large to fully reduce; showing last-pass output."). |
| MCP client disconnects mid-job | Bridge detects the dropped MCP transport, signals the job-level `AbortController`. Each in-flight chunk's call signal is a chain of `[jobSignal, perCallTimeout]`, so the job abort propagates through. Pending p-limit queue items check `jobSignal.aborted` at entry and bail without invoking the backend, avoiding useless drain work. The orchestrator's catch block re-throws on `jobSignal.aborted` (clean exit) and only substitutes placeholders on local timeouts. |
| Source URI read fails | Same as existing tools — `isError: true` with the reader's message. No chunking attempted. |
| `@langchain/textsplitters` breaks change during install | Pinned in package.json. Smoke test T9 is the early-warning signal. |
| js-tiktoken undercounts Qwen tokens by > 20 % | Safety factor: target chunk size × 0.85 when measured in js-tiktoken. Revisit once `diag-long-input.mjs`-style measurements with the new tool exist. v0.3.0 LlamaCppBackend will report exact counts via `/tokenize`. |
| Synchronous tokenization blocks the Node event loop, MCP keep-alive misses | `OllamaBackend.countTokens()` yields to the event loop (`setImmediate`) every ≤ 10 ms of work. Fast-path uses a cheap byte-count estimate first, calling the tokenizer only when in the gray zone. |
| Per-call 50 s timeout aborts the whole job | Each call uses `AbortSignal.any([jobSignal, AbortSignal.timeout(50_000)])` so a local timeout aborts ONLY that call. Catch logic checks `jobSignal.aborted` to distinguish timeout (substitute placeholder) from job cancellation (re-throw). |
| `OMCP_CHUNK_OVERLAP ≥ OMCP_CHUNK_SIZE` | Hard validation at startup before any work. Error message names the offending vars and shows the constraint. Without this `RecursiveCharacterTextSplitter` either throws an unhandled exception or infinite-loops, depending on internals. |

## 8. Testing strategy

**Unit tests (vitest, no Ollama required — runs in CI):**

1. `split.test.ts` — chunk boundary behavior: recursion through paragraph → sentence → character; overlap correctness; empty-input edge cases; single-chunk-fits case.
2. `map-reduce.test.ts` — feed the orchestrator a `FakeBackend` with pre-scripted responses. Assert: correct chunk count, correct prompt construction, correct reduce invocation, partial-result handling on simulated timeout, recursion on oversized reduce, max-depth cap, max_chunks gate, AbortSignal propagation.
3. `migration-snapshot.test.ts` — **new, added per auditor feedback.** For each of the five existing tools: call the handler with a capturing `RecorderBackend`, snapshot:
    - **Backend construction args** — `{ model, keepAlive }` — to catch regressions where migration drops `keepAlive` (which is bound at construction time and would NOT appear in per-call payloads).
    - **Per-call request payload** — `{ system, user, temperature, format, maxInputTokens, maxOutputTokens, stopSequences }`.
    - **Response-handling invariants** — given a fixed `ChatResult`, assert the assembled `_meta` has the right shape (`promptTokens`, `completionTokens`, `model`, `tier`, `defender/*` fields), the footer string is byte-identical, the defender ran in the right order relative to the backend call, and any `schemaStripped`/`schemaValidation` fields land in `_meta` as before.
    - LLM outputs themselves are stochastic and not snapshot-safe; everything else above IS deterministic.
    - A pre-migration commit records the snapshots; post-migration must match byte-for-byte across all 51 smoke cases.
4. `abort-propagation.test.ts` — **new.** Verifies: (a) job-level abort during MAP causes pending p-limit queue items to early-exit without backend calls, (b) per-call 50 s timeout substitutes the placeholder for that chunk only, (c) `jobSignal.aborted` re-throws cleanly without substitution, (d) `AbortSignal.any` chaining is correct (per-call timeout doesn't fire job signal, job signal does fire per-call signals).
5. Existing `sanitize.test.ts` — unchanged.

**Integration tests (require Ollama, excluded from CI):**

1. `smoke-bridge.mjs` — full suite must pass unchanged (behavior preservation of the migration).
2. New T9 in smoke-bridge.mjs — `summarize-long-chunked` round-trip on a medium input (~5000 tokens / 2-3 chunks) asserting: non-error result, footer contains `chunks=`, `_meta.chunks_processed ≥ 2`.
3. `tests/diag-long-input.mjs` (existing) — re-run and demonstrate that the 55 K-token SRT now produces a full-document summary (no left-truncation symptom).

**Manual acceptance (concrete criterion):**

Run the new tool against `/Users/rd/Downloads/S57…SRT` (55 K tokens) from Claude Code. Assert ALL of:

1. **No MCP timeout** — the tool returns a result from Claude Code without client-side config changes.
2. **Full-document coverage via term presence.** The summary output must mention at least one concrete term from EACH of three thirds of the source:
   - **Early third (subtitles 473–700, the episode's title theme):** one of `向上管理` / `角色分工` / `PM` / `架构师` / `设计师`.
   - **Middle third (subtitles 700–1000, the AI-tooling discussion):** one of `MCP` / `Claude Code` / `Codex` / `superpowers` / `memory palace`.
   - **Late third (subtitles 1000–end, the quant-investing / 对口型 section):** one of `巴菲特` / `标普` / `对口型` / `向量知识库`.

   **Strength of this gate.** This is a term-presence check, deliberately pragmatic — it catches the worst failure modes (the chunker drops a third of the source, the reducer collapses content away) but does not certify summary quality. A more rigorous gate (embedding-cosine coverage ≥ 0.75 per third + length-ratio caps + LLM-as-judge factuality) is left to a later release. Term-presence is the v0.2.0 ship criterion; quality-of-summary is observed but not gated.
3. **Chunk count reasonable** — footer shows `chunks=` between 20 and 40 for the target file at default chunk size.
4. **No orphaned Ollama processes** after the test — verified via `ps aux | grep ollama` after running and then simulating a client disconnect mid-job.

## 9. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Reduce prompt still produces mirror-the-chunks output | Medium | Medium (summary quality degrades to a list of chunk summaries) | REDUCE_SYSTEM explicitly instructs "do not mirror" — a lesson already learned in v0.1.2 F4 regression fix. Add T9b assertion: output bullet count ≤ 6 even with 30 chunks. |
| Concurrency=2 is the wrong ceiling on some hardware | Low | Low (perf only) | Env-var override. Not a correctness risk. |
| Backend abstraction leaks Ollama assumptions (e.g., a later tool needs `keepAlive`) | Low-Medium | Medium (might force interface change) | Migration of five existing tools is the stress test. If any tool genuinely needs an escape hatch, the interface gets a `backendHint?: Record<string,unknown>` passthrough — NOT resolved now, but reserved. |

Verified against published `package.json` files at langchain-ai/langchainjs, dqbd/tiktoken, sindresorhus/p-limit during the Pass 4-redo audit.

**Considered and rejected: `p-queue`.** Sindre Sorhus's `p-queue` is the bigger sibling of `p-limit` — adds priority scheduling, pause/resume, throttling, etc. For our use case (bounded fan-out + abort propagation) `p-limit` is sufficient and the explicit `if (jobSignal.aborted) throw` guard at chunk-function entry is small and clear. Adding `p-queue` would expand surface area without unlocking new capability for this scope. Re-evaluate if v0.3.0 introduces real prioritization needs.

**Build-complexity note:** an adversarial review raised the concern that `js-tiktoken` might use WASM / postinstall scripts and complicate a `tsc`-only build. **Verified false**: `js-tiktoken` is explicitly a "pure JS port" of tiktoken (source: [its GitHub README](https://github.com/dqbd/tiktoken/tree/main/js)) — no WASM, no postinstall, zero native components. The other package, `tiktoken`, does use WASM; we are not using that one. `@langchain/textsplitters` depends only on `js-tiktoken`. No build-complexity increase.

## 12. Open decisions for Auditor

Points I believe are settled but want explicit sign-off on:

1. **Tool name.** `summarize-long-chunked` (descriptive, ugly) vs `summarize-chunked` (shorter) vs `summarize-long` v2 (overloads existing). My vote: `summarize-long-chunked` — clearest contract, no silent behavior change.
2. **Chunk size default 2000 tokens.** Empirical per cognitivetech. Alternative: 4000 (fewer chunks, longer per-call, might avoid recursion at target size). My vote: 2000 with env-var override; the project targets summary *quality* over minimum wall time, and the recursive-reduce path is correct to exercise at the target size.
3. **Migration of existing five tools in the same release.** Alternative: migrate only `summarize-long` and defer others. My vote: all five together — the interface is only proven when it's carrying all the load, and splitting the migration creates two inconsistent worlds in the codebase. Guarded by the `migration-snapshot.test.ts` request-payload snapshot.
4. **Max recursion depth 3.** Above this, return partial. Alternative: unlimited. My vote: 3 — if a document needs deeper reduction it's an outlier that deserves a dedicated tool or manual pre-chunking.
5. **No streaming.** Deferred to a later release. OK to leave out of v0.2.0.

### 13.3 Dependencies and peer-deps

- Two runtime additions: `@langchain/textsplitters` (MIT) and `p-limit` (MIT). Pin to caret-minor at install time (`^X.Y.0`) — not exact-version, but bounded.
- No `peerDependencies` declared. The bridge runs as a standalone MCP server; consumers install the bin, they do not embed the library, so peer-dep conflicts with other MCP servers in the same machine are not a concern (each MCP server runs in its own subprocess).

### 13.4 Pre-publish checks

Add to `package.json` `scripts`:

```jsonc
{
  "scripts": {
    "prepublishOnly": "npm run build && npm test && node tests/check-version-sync.mjs"
  }
}
```

`check-version-sync.mjs` (small new helper): asserts the version in `package.json` matches the latest entry in `CHANGELOG.md`. Prevents shipping an unmarked release.

### 13.5 Deprecation policy

No deprecations in v0.2.0. The `summarize-long` tool stays alive next to `summarize-long-chunked` — they have different contracts (single-call vs map-reduce) and clients should choose deliberately. Re-evaluate consolidation in v0.3.0 once chunked summarization has real-world data.
