## 1. What this release delivers

Two concerns bundled because they target the same root cause — the bridge is
not actually being used to save frontier tokens during real work.

1. **Async job pattern + persistent Memory Bank** — three new MCP tools
   (`enqueue-job`, `wait_for_job`, `read_job_result`) that let frontier
   clients dispatch long-running work (e.g. `summarize-long-chunked` on a
   big SRT) without hitting Claude Code's hardcoded 60 s MCP wall, with the
   result persisted to disk so subsequent sessions can pick it up.

2. **`diff-semantic-index` tool + `.claude/commands/draft-from-diff` hook** —
   a structured-extract tool that turns a `git diff` into a typed JSON
   document; a user-side hook composes that JSON into a commit-message draft
   the frontier polishes. Local model produces structure, frontier writes
   prose.

These bundle because (a) both target frontier-token bottlenecks documented
in the v0.3.0 decision (research reading vs. doc writing), and (b) shipping
both at once gives the user concrete proof points for whether the bridge
actually moves the needle.

## 2. Out of scope for v0.3.0

- **`llama.cpp` backend swap** — moved to v0.4.0 per the decision doc.
- **Push-based job completion (webhook)** — long-poll first; can add push
  in v0.4.0 if real usage shows long-poll bottlenecking.
- **Multi-pass local critique orchestrator** — explicitly rejected (Copilot
  proposal #3 in the brainstorm). Re-litigation requires new evidence.
- **Auto-routing of existing tools through the job pattern** — opt-in only.
  Existing 6 tools keep their direct-call contracts unchanged.
- **Cross-session job continuation** — if the bridge restarts mid-job, that
  job is marked failed; we do not attempt to resume mid-flight work.
- **Non-`file://` source URIs in the diff hook** — the hook reads `git diff`
  output as inline text, not via `source_uri`.

## 3. New MCP tools

### 3.1 `enqueue-job`

```
enqueue-job(
  tool_name:  string,                  // one of the v0.2.0 tools (not enqueue-job itself)
  args:       Record<string, unknown>, // forwarded to the underlying tool
  ttl_days?:  number = 7,              // memory bank retention
)
→
{
  job_id: string,                       // nanoid(10), used as path-safe filename
  enqueued_at: string,                  // ISO8601
  expires_at: string,                   // ISO8601, mtime+ttl_days
}
```

- Validates `tool_name` against the registered tool list. Rejects with
  `isError: true` if unknown.
- Validates `args` against the target tool's input schema using the existing
  Zod schemas. Rejects with `isError: true` on mismatch.
- Generates `job_id = nanoid(10)`. Writes `.memory/jobs/<id>.json` immediately
  with `status: 'queued'` and the request payload.
- Returns within < 100 ms — never blocks on the actual work.
- Schedules the work via `p-queue` (concurrency = `OMCP_JOB_CONCURRENCY`,
  default 1). Worker writes status updates as the job progresses.

### 3.2 `wait_for_job`

```
wait_for_job(
  job_id:        string,
  max_wait_ms?:  number = 50000,       // capped at 55 000 to leave MCP-wall margin
)
→
{ status: 'running',  progress?: { current: number, total: number, message: string } }
| { status: 'done',     result_path: string, footer: string }
| { status: 'failed',   error: string }
| { status: 'unknown',  reason: 'expired' | 'never_existed' }
```

- Implements RFC 6202 long-polling. Blocks up to `max_wait_ms` waiting for
  the job's status to become `done` or `failed`. Returns immediately on
  state change. Cap of 55 s ensures we never hit the 60 s MCP wall.
- The frontier client calls this in a loop (or once), each call within budget.
  This is the hand-off pattern that closes the 60 s wall.
- Internally: subscribes to the in-memory job-registry event emitter for
  the job_id; falls back to mtime-poll on the `.json` sidecar (1 s interval)
  if the registry doesn't have the job (e.g., bridge restart).

### 3.3 `read_job_result`

```
read_job_result(
  job_id: string,
)
→ string                                 // the .md result body
| { isError: true, content: [{type: 'text', text: '...'}] }
```

- Reads `.memory/jobs/<id>.md`.
- Returns `isError` for: not found, expired, status not `done`.
- Footer (the `[bridge:`-prefixed line) is included in the body so the
  frontier sees the same telemetry as a synchronous call.

### 3.4 `diff-semantic-index`

```
diff-semantic-index(
  diff_text: string,                  // raw `git diff` output
)
→
{
  change_type: 'feature' | 'fix' | 'refactor' | 'docs' | 'test' | 'chore' | 'mixed',
  summary: string,                     // 1-sentence overview
  files_touched: Array<{
    path: string,
    role: 'added' | 'modified' | 'deleted' | 'renamed',
    lines_changed: number,
    role_in_change?: string             // 1-line of why this file changed
  }>,
  key_decisions: string[],              // 3-7 architectural points visible in the diff
  risk_callouts: string[],              // potential regressions / breaking changes
  test_coverage_hint: 'tests_added' | 'tests_modified' | 'no_test_change' | 'unclear'
}
```

- Routes to **Tier B** (`qwen3:4b`). Structured output, modest reasoning;
  fast model is right.
- Uses `parse-diff` to pre-structure the input — gives the model a clean
  list of hunks with paths and line numbers instead of raw text.
- Output grammar-constrained via the existing Ollama `format:` mechanism
  (same path as `extract`).
- Defender (Tier-1 always-on) runs on `diff_text`. Spotlighting wrapper
  applied as with other text-input tools.

## 6. Failure modes and mitigations

| Failure | Mitigation |
|---|---|
| Bridge crashes mid-job | On startup, scan `.memory/jobs/*.json` for `status: 'running'`; mark them `failed` with `error: 'bridge restarted'`. The .md may be partial; truncate to last newline. |
| FS write race (two writes to same job_id) | Atomic write: `fs.writeFile(tmp); fs.rename(tmp, final)`. Job IDs are nanoid(10) → ~58 bits of entropy; collision rate negligible. |
| Memory Bank disk usage unbounded | TTL = 7 d (default; configurable via `OMCP_MEMORY_TTL_DAYS`). GC sweep on every bridge startup. Manual `OMCP_MEMORY_GC_NOW=1` env var triggers immediate sweep. |
| User's `.memory/` in `.gitignore`? | Document in README that `.memory/` should be gitignored. Add `.gitignore` entry to the bridge's own repo as example. |
| `wait_for_job` called with bad `job_id` | Returns `{ status: 'unknown', reason: 'never_existed' }`. Idempotent; safe to retry. |
| Job's underlying tool throws during execution | Caught in runner; `.json` updated to `failed` with stack trace. `read_job_result` returns `isError` with the trace. |
| Diff too big for Tier B's `num_ctx = 8192` | `diff-semantic-index` checks `diff_text` length first. If estimated tokens > 7 K, returns `isError` with hint to use `enqueue-job` + `summarize-long-chunked` first to shrink the diff. **Does not auto-chunk** — that would change the contract semantics. |
| Diff contains binary file changes | `parse-diff` reports them as files with no hunks; we surface them in `files_touched` with `lines_changed: 0` but skip semantic indexing of their bytes. |
| AbortSignal from MCP client during `wait_for_job` | The wait returns immediately. The underlying job continues. The client can re-attach via `wait_for_job(same_id)` or `read_job_result` later. Job state is the source of truth, not the wait call. |
| Local model produces invalid JSON for `diff-semantic-index` | Grammar-constrained output via Ollama `format:` schema makes structurally invalid output near-impossible (same mechanism as v0.2.0 `extract`). If it still fails, return `isError`. |

## 7. Testing strategy

**Unit tests (vitest, no Ollama, runs in CI):**

1. `job-store.test.ts` — write/read round-trip; TTL boundary (mtime older
   than threshold marked expired); GC sweep removes expired files; atomic
   write doesn't leave partials on simulated crash; concurrent writes serialize.
2. `job-registry.test.ts` — enqueue → status='queued' → 'running' → 'done'
   transitions emit events; same id can be subscribed by multiple waiters;
   on registry-restart, persisted-but-not-in-memory jobs surface via fallback
   poll.
3. `diff-parse.test.ts` — fixtures: clean diff (one file modified), multi-file
   diff, diff with binary files, diff with renamed files, diff with deleted
   files. parse-diff output → our typed object.
4. `diff-index.test.ts` — uses FakeBackend with scripted JSON responses;
   verifies prompt construction; verifies grammar schema is the documented
   one; verifies parse-diff output is included in user prompt; verifies
   `change_type` enum gets enforced.

**Migration snapshot:** unchanged from v0.2.0. The 5 existing tool handlers
do not change.

**Integration tests (smoke, requires Ollama, excluded from CI):**

1. **T10 — async job round-trip.** `enqueue-job(summarize, {text: "..."})`
   → `wait_for_job` returns running, then done → `read_job_result` returns
   the same content as a direct `summarize` call would. Asserts `job_id`
   format, `status` lifecycle, footer present in result, `.memory/jobs/<id>.md`
   exists post-run.
2. **T11 — diff-semantic-index.** A fixed-content diff fixture (committed to
   repo) → `diff-semantic-index(diff)` → assert returned JSON has the right
   shape, `change_type` is one of the enum, `files_touched.length > 0`, no
   `risk_callouts.length` errors.

Total smoke after v0.3.0: **60 checks** (was 58).

## 9. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Frontier polls `wait_for_job` too aggressively, burns turns | Medium | Medium | Tool description states "call once with `max_wait_ms` ≥ 30 s, not in tight loop". Document in CLAUDE.md. Cap server-side at 55 s regardless of caller value. |
| Disk fills up with stale .memory/ | Low | Medium | TTL + startup GC. Default 7 d × typical job size 5-50 KB × maybe 100 jobs = < 5 MB. |
| Bridge restart during a job mid-MAP-phase loses work | Low | Low | Job marked failed; user re-enqueues. Future v0.4.0+ could persist intermediate chunks; not in scope. |
| `parse-diff` choking on edge-case diffs (submodules, complex renames) | Low | Low | Try-catch around parse-diff; on failure send raw diff_text as fallback (model still gets the data, just less structured). |
| `diff-semantic-index` for huge diffs blows Tier B context | Low | Medium | Hard error with hint, not silent truncation. User can pre-chunk via existing chunked tool. |
| `.claude/commands/draft-from-diff` doesn't work in Cursor / other clients | Medium | Low | Hook is documented as Claude-Code-specific. Docs note it can be ported (the bridge tool is client-neutral). |
| User confusion: which tool to use? `summarize-long-chunked` vs `enqueue-job summarize-long-chunked`? | High | Low | Tool descriptions explain: use the direct tool if you can wait < 60 s; use `enqueue-job` otherwise. CLAUDE.md prescribes the rule. |

## 10. Rollback plan

The 10 commits above are sequenced for clean bisect. Each commit:
- Adds new code (no replacement of existing logic).
- Keeps the smoke suite passing with all v0.2.0 tests green.
- Has its own rollback story: `git revert <sha>` removes that step's
  contribution, leaves prior steps intact.

Worst-case: revert all 10 commits (one revert range); v0.2.0 behavior
restored exactly. The migration-snapshot test guards against any silent
contract change to the existing 5 tools.

## 12. Open decisions for Auditor

Points I believe are settled but want explicit sign-off on:

1. **Long-poll vs push.** Going long-poll. PA #1 recommended; decision doc
   confirmed; this memo confirms. Easier to add push in v0.4.0 if needed
   than to remove it.
2. **Job concurrency = 1 default.** Ollama serializes on Metal anyway;
   higher concurrency just adds queueing overhead for no throughput gain.
   Configurable via `OMCP_JOB_CONCURRENCY`. OK?
3. **TTL = 7 days default.** Configurable. Long enough for "summarize this
   research, I'll come back to it next session"; short enough that disk
   stays bounded. OK?
4. **`.memory/` at user's project root.** Not OS temp (would be wiped on
   reboot, defeating "persistent across sessions"). Configurable via
   `OMCP_MEMORY_DIR`. OK?
5. **`diff-semantic-index` routes to Tier B.** Diffs are short, structure
   is more important than reasoning depth. Tier B is faster + cheaper.
   Reasonable?
6. **Existing tools NOT auto-routed through async.** Opt-in only via
   `enqueue-job(toolName, args)`. Rationale: don't break v0.2.0's contracts.
7. **`.gitignore` entry for `.memory/`.** Bridge's own repo gets one as
   example; users decide for their own repos.
8. **Tool-name validation in `enqueue-job`.** Whitelist the 6 v0.2.0 tools
   (not `enqueue-job` itself, not `wait_for_job`, not `read_job_result`).
   Pure name match.
