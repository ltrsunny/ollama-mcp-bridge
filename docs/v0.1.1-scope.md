### Unknown #3 — is Ollama's native `format: <json-schema>` reliable enough on Tier B for an `extract` tool?

**Within a narrowed schema regime, yes.** Evidence split across two runs.

**Run A — 150 benign cases** (`/tmp/extract-eval/results.json`, `qwen3:4b-instruct-2507-q4_K_M`, temp 0.2):

| Tier | Schema shape | Pass rate | p50 latency | p95 latency |
|---|---|---|---|---|
| easy | flat 2 fields | 50/50 = 100% | 691 ms | 738 ms |
| medium | enum + array of strings | 50/50 = 100% | 1781 ms | 1903 ms |
| hard | nested 2-level + array of objects | 50/50 = 100% | 3608 ms | 4607 ms |

Spot-checked 5 hard cases by hand for semantic correctness (names/emails/SKUs/qty/prices/totals all reproduced faithfully).

**Run B — 20-case adversarial v2** (`/tmp/extract-eval/results-adversarial-v2.json`, same model, temp 0.2 + 0.8 mix):

| Shape | Pass rate | Median latency |
|---|---|---|
| `anyOf` union (string OR object) | 4/4 structural / **2/4 semantic¹** | ~1.9 s |
| array-of-arrays (numeric grid) | 4/4 | ~1.5 s |
| length bounds (`minLength`/`maxLength`, `minItems`/`maxItems`) | 4/4 | ~1.0 s |
| Chinese-only prompt body + Chinese names | 4/4 | ~1.2 s |
| 3-level deep nested (company → depts → teams → members) | 4/4 | ~3.7 s |

**¹ anyOf semantic caveat** — Zod `safeParse` against `z.union([z.string(), z.object({tag, score})])` accepts either branch, so all 4 cases pass *structurally*. On independent semantic review of the raw outputs (post-hoc, 2026-04-23 Auditor v2 Check 2), **2 of 4 cases the model routed plain-string items into the object branch with fabricated `score: 0`** (i=0 temp=0.8 Chinese "产品迭代": 2/3 items mis-routed; i=2 temp=0.8 "Survey": 3/3 items mis-routed). Structural grammar enforcement does **not** imply branch-selection fidelity for bare `z.union`. Ship guidance: F2 tool docstring directs callers to `z.discriminatedUnion` where possible.

**Run C — discovered failure mode** (same probe, pre-v2 run, reproducible):

Schemas containing `"pattern": "..."`, `"format": "email"`, or `"format": "uri"` (typical output of Zod `z.email()` / `z.url()` / `z.string().regex(...)`) **crash the Ollama runner process** with `model runner has unexpectedly stopped` → subsequent requests return `ECONNREFUSED 127.0.0.1:11434` until the daemon respawns a new runner. Root cause: llama.cpp's GBNF grammar compiler cannot handle these constraints and the runner aborts without returning an error payload. This is **not a model problem; it's an Ollama/llama.cpp limitation** that any caller using Zod must guard against.

**Tested regime Unknown #3 covers (claim is bounded here)**:
- ✓ Model: `qwen3:4b-instruct-2507-q4_K_M` only.
- ✓ Temperature: 0.2 (150 cases) + 0.8 (15/20 adversarial cases).
- ✓ Schema shapes: flat objects, arrays of primitives, arrays of objects, enums, `anyOf` unions of string/object (**structural validity only — see ¹ under Run B**), arrays-of-arrays, 3-level nesting, minLength/maxLength/minItems/maxItems.
- ✓ Language: English primarily; **7 adversarial Chinese prompts** (4 in the dedicated `chinese-body` case + 3 incidental in `anyOf-union` / `array-of-arrays` / `length-bounds`) + CJK-name injection in ~20 of the 150 benign.
- ✗ **Not** tested: other Tier-B models (dropping qwen3 for llama3.x etc.), temp ≥ 1.0, schemas with `pattern`/`format: email`/`format: uri` (known-crash, see Run C), schemas with circular `$ref`, schemas with numeric `multipleOf` or `minimum`/`maximum` bounds, schemas >10KB JSON, prompts >4K tokens, **`anyOf` branch-selection semantic fidelity for bare `z.union`** (Run B ¹; structurally validated but the model picks the wrong branch in ~50% of the adversarial cases when branches have disjoint shapes).

**Narrowed ship decision**:
1. **F2 `extract` ships with a `sanitizeSchemaForOllama()` preprocessor** in the bridge. It walks the JSON schema and strips: `pattern`, `format: email`, `format: uri`, `format: date-time` (assume untested = risky), numeric `multipleOf`. Stripped constraints are returned to the caller in `_meta` so the caller can re-validate via Zod on the bridge's output.
2. **safeParse-on-output** remains: if after schema sanitation the model still emits something that fails the caller's full Zod schema, we return `isError: true` with raw output + stripped-constraints list, letting the caller decide (retry with smaller schema, split the payload, skip).
3. **No in-bridge retry loop** for v0.1.1. If users hit retry-worthy failures in the wild we'll add it in v0.1.2.

## v0.1.1 feature list

### F1 — `classify` tool

- **Route**: Tier B.
- **Contract**: `classify(text, categories: string[], allow_multiple?: boolean, explain?: boolean) → { labels: string[], reason?: string }`.
- **Mechanism**: `format:` = JSON schema with `labels: array of enum(categories)` (+ `minItems: 1` / `maxItems: 1` when `allow_multiple === false`). Grammar guarantees labels are valid members.
- **Language preservation**: system prompt explicitly tells model to preserve source language in `reason` when `explain === true`.

### F2 — `extract` tool

- **Route**: Tier B.
- **Contract**: `extract(text, schema: JSONSchemaObject) → { data: <schema-typed> } | isError`.
- **Schema guidance (surfaced in tool description)**: prefer **`z.discriminatedUnion`** over bare `z.union` when branches overlap on output shape. Unknown #3 Run B ¹ showed bare `z.union([string, object])` passes structural grammar but the model routes ~50% of items into the wrong branch on adversarial prompts; a discriminator field collapses that ambiguity. Non-union schemas are unaffected.
- **Bridge-side schema sanitation**: a `sanitizeSchemaForOllama(schema)` preprocessor walks the input and **strips** (with `_meta` attribution of what was stripped):
  - `pattern` on any string
  - `format: "email" | "uri" | "date-time"`
  - `multipleOf` on numbers
  - `$ref` (we refuse and return `isError` — resolving refs is out-of-scope for v0.1.1)
  This preserves grammar-compiler stability (Run C). Stripped constraints are surfaced to the caller via `_meta[dev.ollamamcpbridge/schema_stripped]` so caller-side validation (e.g. Zod `safeParse`) can re-apply them if desired.
- **Token cap**: `num_predict: 2048` to bound hostile-schema generation.
- **safeParse fallback**: if the caller passed a Zod schema and the model output fails full Zod validation (not just the sanitized subset), return `isError: true` with `content: [{type:'text', text: rawOutput}]` and `_meta[dev.ollamamcpbridge/schema_validation]: 'failed'`.
- **No in-bridge retry loop** for v0.1.1.

### F3 — `transform` tool

- **Route**: Tier B.
- **Contract**: `transform(text, instruction) → string`.
- **Mechanism**: plain chat, no `format:`. System prompt: "Apply the instruction to the text. Return ONLY the transformed text. Preserve the source language unless the instruction explicitly says otherwise."

### F4 — prompt-injection defense (opt-in per tool)

- **Layer 1a — Spotlighting (delimiting)**: wrap untrusted user text in a **unique per-call delimiter token** announced in the system prompt. Source: arxiv 2403.14720, §3 ("Delimiting"). ~10 LOC inline.
- **Layer 1b — Unicode NFKC normalization**: `text.normalize('NFKC')` on untrusted text before delimiting. Source: Unicode TR#15 (https://unicode.org/reports/tr15/). **Separate** technique from Spotlighting — labeled and documented as such.
- **Layer 2 — `@stackone/defender`**: call `defendToolResult(text, toolName)` on tool inputs carrying external text. On `!allowed` we return `isError: true` with `_meta[.../defender/*]` attribution and do **not** call Ollama. Defaults to **Tier-1 only** (~24 MB); Tier-2 MiniLM enabled via `OMCP_DEFENDER_TIER2=1`.
- **Warmup**: if Tier-2 enabled, `warmupTier2()` is called at bridge boot so the 1.6 s load doesn't surprise the first tool call.
- **Opt-in per tool**: config key `defendUntrusted: boolean`; default `true` for `summarize`, `summarize-long`, `classify`, `extract`, `transform`. Callers can set `false` for trusted internal text.

### F5 — `_meta` emission

Keys listed above under "Prior art per feature → F5". Implementation is a small `buildMeta()` helper called by every tool handler; always-emitted keys come from the Ollama response (`prompt_eval_count`, `eval_count`) + the resolved tier config; conditional keys are set when the defender or schema-sanitizer ran.

### F6 — streaming progress notifications

- When the host supplies `_meta.progressToken` in the tool call, bridge calls `extra.sendNotification({method: 'notifications/progress', params: {progressToken, progress, total, message}})` at three waypoints:
  - `0/3 — routing to Tier B (<model>)`
  - `1/3 — defender checked (risk=<level>)` — omitted if `defendUntrusted: false`
  - `2/3 — Ollama generating (<eval_count_so_far> tokens)` — piggy-back on Ollama's chunked response
- When no progressToken: skip silently, no behavior change.

## Out-of-scope for v0.1.1

- Tool discovery / dynamic model pulling — users still `ollama pull` manually.
- Non-stdio transports (SSE, HTTP) — stdio only.
- Model catalog as an MCP resource — keep as CLI.
- Dynamic agent-count throttling based on CPU/MEM — deferred, flagged as "measure real need first".
- Non-English/Chinese language coverage beyond what qwen3 already does.
- Custom classifier retraining for F4 Layer 2 — ship with defender's bundled MiniLM.
- OTel span emission — v0.1.2 candidate, separate from `_meta`.
- Schema sanitizer for `$ref` resolution — caller must pre-flatten.

## Implementation order

1. **F5 `_meta` scaffold** first (small, cross-cutting). Update `ollama/client.ts` return shape to expose `prompt_eval_count`, `eval_count`, `total_duration`. Add `buildMeta()` helper in new `src/mcp/meta.ts`.
2. **F4 Layer 1 inline helpers** (spotlighting + NFKC) as pure functions in `src/security/spotlight.ts`. **F4 Layer 2 wrapper** (`src/security/defender.ts`) with Tier-2 opt-in detection.
3. **F2 schema-sanitizer** `src/mcp/extract/sanitize.ts` — pure function, unit tests required.
4. **F1 classify** — simplest `format:` path, validates enum-constrained schema end-to-end.
5. **F2 extract** — uses the sanitizer + full-schema safeParse fallback.
6. **F3 transform** — no `format:`, last.
7. **F6 progress** — cross-cutting, added after all handlers exist.

## Risks

1. **Schema sanitizer false-negative** — if a Zod schema nests a regex-constrained string 5 levels deep, our walker must catch it. Unit tests must cover nested cases (object inside array inside object). Failure mode = runner crash = daemon restart = bad UX.
2. **Defender Tier-1 false positives on legitimate text containing injection-like phrasing** ("ignore the previous paragraph"). Document `defendUntrusted: false` opt-out prominently.
3. **Bundle-size opt-in UX** — users enabling Tier-2 see a 20x size jump; document prominently in README. Tier-1 should cover >80% of real-world attacks (regex-based, per defender's own docs).
4. **Progress notification ordering** — `sendNotification` is async; if the tool handler returns before the notification flushes, the host might receive them out-of-order. Verify in smoke test that all 3 waypoints arrive before the result.
5. **Schema-sanitizer stripped-constraint attribution** — caller may think bridge enforced the regex when it didn't. `_meta[.../schema_stripped]` must be prominent in tool description so host LLMs know to re-validate.
