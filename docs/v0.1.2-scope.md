# v0.1.2 scope — ollama-mcp-bridge

Status: **Approved** — Auditor v2 pass: Claude Sonnet 4.6 inline, 2026-04-24. Implementation in progress.
Date: 2026-04-23.
Predecessor: v0.1.0 shipped summarize/summarize-long; v0.1.1 shipped classify/extract/transform + `_meta` + injection defense + progress notifications.

## Theme

**"Prove it saves tokens"** — v0.1.1 delivered 5 tools and observability plumbing, but first-hand use revealed the bridge **does not actually save tokens in its current architecture**, and the frontier has no way to see whether it's saving anything. v0.1.2 closes both gaps before any further feature expansion.

## Pre-memo: first-hand usage observations (2026-04-23)

The frontier LLM working on this codebase (me) was the first and only user of v0.1.1. Five delegated calls against real-session content produced:

| Tool | Input | Quality | Notes |
|---|---|---|---|
| `summarize` (B) | 230 tok | 6/10 | Specifics lost ("产品迭代" → "Chinese-language prompts"); verbose |
| `classify` (B) | 30 tok | **2/10** | Mis-labeled a correction as "approval"; `reason` field echoed input verbatim |
| `extract` (B) | 170 tok | **10/10** | Grammar-constrained JSON is the sweet spot — real arithmetic (214+261=475; 1.6s→1600ms) |
| `transform` (B) | 20 tok | 7/10 | Slight expansion ("invaluable" added) |
| `summarize-long` (C) | ~2000 tok | 5/10 | Output 20 bullets vs promised 3-6; largely caused by bullet-structured input (mirroring) |

**Meta-observations, more serious than any single tool's quality:**

1. **`_meta` is invisible to the calling LLM.** Every call returns only `content` text; latency/tokens/defender telemetry are silently dropped by the client. The observability story built in v0.1.1 is consumed by humans on dashboards but not by the frontier making delegation decisions. **There is no feedback loop.**
2. **Tool-use arguments count as output tokens.** If the source text is already in the frontier's context and gets passed to the bridge as a tool argument, the frontier **pays output-token cost for the full source**, then pays input-token cost for the summary on the next turn. Delegation via the current API is **more expensive** than summarizing in-line, not less.
3. **Tools live in the "deferred" set.** I have to `ToolSearch` to load each schema before I can call it. Friction high enough that even I — the implementer — didn't think to invoke them until explicitly prompted.

## Decisions locked before build

Four unknowns closed prior to scoping:

### Unknown A — `_meta` visibility to frontier LLM

Investigation: MCP spec defines `_meta` on `CallToolResult` for client/app consumption (OTel exports, billing, dashboards) — **not** for inclusion in the LLM-visible portion of the tool result. The frontier only sees `content[]`.

**Closed**: telemetry visible to the frontier must live inside `content[]`. Preserve `_meta` for machine consumers (unchanged from v0.1.1) but also emit a terse text footer as the **last element** of `content[]`. Host apps that don't want the noise can strip structured-content items of a known type before showing to users.

### Unknown B — Ollama default `num_ctx`

Investigation: sent a 6000-token probe to `qwen3:4b-instruct-2507-q4_K_M` without setting `num_ctx`. Response `prompt_eval_count = 4096`; Ollama silently **left-truncates** (drops the beginning, keeps the end). The model supports 262 144-token context per its modelfile, but Ollama's runtime default is 4096 unless set.

**Impact**: summarize-long's "~2000+ words" contract violates silently for English inputs over ~3000 words; extract on a large JSON source produces hallucinations (missing opening context).

**Closed**: set `num_ctx` explicitly per tier. Tier B → 8192, Tier C → 16384 (provisional; VRAM-benchmarked during implementation on 16 GB Mac baseline).

### Unknown C — classify `reason` field design bug

Investigation: read `CLASSIFY_SYSTEM` in `server.ts`. Prompt currently says `"Preserve the source language inside the reason field."` but **never specifies that `reason` should contain an explanation**. Model correctly interprets "preserve the source language" as "copy the source text" and echoes the input.

**Closed**: prompt surgery, one line change:
> "If a `reason` field is requested, write ONE brief sentence explaining why you chose the label(s). Use the same language as the source text."

### Unknown D — where tokens can actually be saved

Following from the meta-observation above: the only regimes where the bridge saves the frontier tokens are:

1. **Bridge reads source directly** (local file, URL) — the source never traverses the frontier's context window; only the output (summary, extracted JSON) returns.
2. **Bridge is called on text that was going to be produced anyway** (a multi-step pipeline where the bridge's output is an intermediate artifact the frontier discards).

Regime 1 is the tractable one. Regime 2 is theoretical and doesn't match real usage patterns.

**Closed**: v0.1.2's central feature is adding file/URL direct-read to heavy-input tools. Without this, the bridge is architecturally incapable of saving tokens. With it, a Claude Desktop user asking "summarize this PDF at /Downloads/x.pdf" can actually save tokens.

## Prior art per feature

### F1 — Explicit `num_ctx` per tier

| Candidate | Verdict | Reason |
|---|---|---|
| Ollama `options.num_ctx` in chat API | **Adopt** | Native, already supported by `ollama-js` v0.6.x. Free config value. |
| Modelfile `PARAMETER num_ctx` override | Reject | Requires distributing a custom modelfile; users can't `ollama pull` stock models. |
| Dynamic per-call sizing based on input length | Reject (v0.1.2) | Adds logic complexity; fixed per-tier is simpler and VRAM-predictable. Consider for v0.1.3. |

### F2 — File/URL input for summarize-long, extract, transform

| Candidate | Verdict | Reason |
|---|---|---|
| Add optional `source_uri` param; bridge reads directly | **Adopt** | Pure addition, backwards compatible, works with any MCP host. Bridge already runs on the user's machine; file access is cheap. |
| MCP `resources` protocol (client passes URI, server registers resource) | Reject | Requires the host to implement MCP resources. Claude Desktop's support is partial. Too client-dependent. |
| Delegate to MCP "filesystem" server upstream, pipe to bridge | Reject | Two-hop, brittle, no atomic semantics. |
| Accept a `file://` or `https://` URI as the `text` field itself | Reject | Confuses the contract. Explicit `source_uri` is better. |

Security model for `source_uri`:
- `file://` reads: no restriction by default. The bridge already runs as the user; the host app's trust boundary covers this.
- `http(s)://` reads: size cap (default 10 MB), timeout (default 30 s), must pass a content-type allowlist (`text/*`, `application/json`, `application/xml`, `application/pdf` deferred — handled via optional `pdf` peer dep or simple rejection in v0.1.2).
- Configurable via `OMCP_URL_MAX_BYTES`, `OMCP_URL_TIMEOUT_MS`.
- **Not** adding a filesystem allowlist (user already trusts what they launched).

### F3 — Content-embedded telemetry footer

| Candidate | Verdict | Reason |
|---|---|---|
| Append terse one-line footer as last `content[]` item | **Adopt** | Visible to frontier; a host app can strip items tagged as telemetry before displaying. ~10 tokens per call. |
| Second `content[]` item typed as `"resource"` or custom annotation | Reject | MCP spec does not define a standard "hide from user but show to LLM" content type; annotations are advisory. |
| No footer; rely on frontier to `ToolSearch` for status | Reject | Adds an extra tool call every time, worse token cost than the footer. |
| Footer opt-out via env `OMCP_TELEMETRY_FOOTER=0` | **Adopt alongside** | Gives users control. Default on. |

Footer format (~10-12 tokens):
```
[bridge: qwen3:4b B 1240ms in=230 out=85 saved~=+210]
```

### F4 — Prompt surgery for `classify.reason` and summarizer tightness

| Candidate | Verdict | Reason |
|---|---|---|
| One-sentence rewrite of `CLASSIFY_SYSTEM` spelling out `reason` semantics | **Adopt** | Root cause of T2 miss in self-observation. |
| Lower `summarize` / `summarize-long` default temp from 0.7 → 0.2 | **Adopt** | Summarization benefits from determinism; Ollama's default leaks through. |
| Tighten `SUMMARIZE_LONG_SYSTEM` to enforce bullet ceiling ("at most 6 bullets; if the input is itself a bullet list, collapse groups") | **Adopt** | Addresses the mirroring failure in T5. |
| Rewrite all five system prompts from scratch | Reject | Unneeded; extract and transform prompts are fine. Surgical > wholesale. |

### F5 — Estimated-savings emission in `_meta` + footer

| Candidate | Verdict | Reason |
|---|---|---|
| Emit `dev.ollamamcpbridge/saved_input_tokens_estimate` = (source tokens forwarded to Ollama) − (frontier-model output token count) | **Adopt** | Rough but useful; makes the savings narrative auditable. Only non-zero when `source_uri` was used. |
| OTel GenAI `gen_ai.usage.*` span attributes | Reject for v0.1.2 | Still too much surface area (scope memo v0.1.1 CHECK 6 reasoning applies — OTel is a separate emission path, not `_meta`). Revisit for v0.1.3. |
| `anthropic-tokens-saved` header-style | Reject | Not a real header; no precedent. |

## v0.1.2 feature list

### F1 — Explicit `num_ctx` per tier

- `src/config/tiers.ts`: add `numCtx: number` to `TierConfig`. Tier B → `8192`, Tier C → `16384`.
- `src/ollama/client.ts`: add `numCtx?: number` to `ChatOptions`; pass through as `options.num_ctx`.
- `src/mcp/server.ts`: pass `tierCfg.numCtx` on every `client.chat` call.
- VRAM benchmark (implementation phase): on baseline 16 GB Mac, log peak VRAM when running `qwen3:4b` at 8 K and `qwen2.5:7b` at 16 K. If either tier OOMs, reduce that tier's cap.

### F2 — File/URL input for heavy tools

- Affected tools: `summarize-long`, `extract`, `transform` (optional), `summarize` (optional for consistency).
- Add optional `source_uri: string` parameter (mutually exclusive with `text`).
- New helper `src/io/sourceReader.ts`:
  - `readSource(uri: string, { maxBytes, timeoutMs })`: dispatches on `file://` or `http(s)://`; returns `{ text: string, bytes: number, contentType: string }`.
  - Rejects unsupported schemes with `isError: true`.
  - `http(s)` reads: `fetch` + AbortController for timeout + content-length/stream-abort for size.
- Injection defense still applies to fetched content (treated as untrusted — in fact more so).
- `_meta` gains `dev.ollamamcpbridge/source_uri` and `dev.ollamamcpbridge/source_bytes` when used.

### F3 — Content-embedded telemetry footer

- `src/mcp/footer.ts`: new helper `buildFooter({ model, tier, latencyMs, promptTokens, completionTokens, savedTokensEstimate? })` → string.
- Every tool handler appends `{ type: 'text', text: buildFooter(...) }` as the **last** element of `content[]`.
- Env opt-out: `OMCP_TELEMETRY_FOOTER=0` disables (still emitted in `_meta`).
- Footer format: `[bridge: <model> <tier> <latency>ms in=<prompt> out=<completion> saved~=+<N>]`. The saved estimate is omitted if `source_uri` was not used.

### F4 — Prompt surgery

**`CLASSIFY_SYSTEM` rewrite** (preserves existing enum-grammar behavior):
> "You are a precise classifier. Given a text and a list of categories, assign the correct label(s) and reply with JSON matching the schema exactly. If a `reason` field is requested, write ONE brief sentence explaining your choice, in the same language as the source text."

**`SUMMARIZE_SYSTEM`** and **`SUMMARIZE_LONG_SYSTEM`**: add explicit temp setting (0.2) at the call site (not in the prompt). Add to long-summary prompt: "If the source is itself bullet-structured, collapse related bullets; never exceed 6 bullets in the output."

**No change** to `EXTRACT_SYSTEM` or `TRANSFORM_SYSTEM` (working correctly in self-observation).

### F5 — Estimated-savings

- Add `savedTokensEstimate?: number` field to the `buildMeta` input.
- Compute in `summarize`, `summarize-long`, `extract`, `transform` handlers **only when `source_uri` was used**:
  - estimated savings = `sourceBytes / 4` (rough token count) − `result.completionTokens`
  - (i.e. what frontier would have paid to pass the source through as output tokens, minus what it pays now to read the bridge's output as input)
- Surfaced in both `_meta[dev.ollamamcpbridge/saved_input_tokens_estimate]` and the footer.
- When `source_uri` was NOT used: omit; the footer shows `saved~=0` or hides the field.

- **Claude Desktop `.mcpb` one-click installer** — same reasoning; packaging is easy, usability is not.
- **GitHub Actions CI** — will be added on Student Pack activation (~2026-04-26) as a separate PR, not a scope line; it's infrastructure, not a user-facing feature.
- **OTel span emission** — deferred again. The `_meta` + footer path is serving the telemetry need for now.
- **In-bridge retry loop for `extract`** — the savings narrative fix is higher priority; retry loop remains v0.1.3 candidate.
- **`$ref` resolution in the schema sanitizer** — still rejected with `isError: true`; caller pre-flattens.
- **New tools** (translate, embed, etc.) — 5 tools is enough; exploit depth before breadth.
- **Non-stdio transports** — stdio still covers 100% of MCP host landscape.
- **Dynamic `num_ctx` based on input length** — fixed per-tier is enough for v0.1.2.
- **Tier-2 defender benchmark** — independent research task; fork into `/tmp/defender-bench/` if pursued, not part of this scope.
- **qwen2.5:14B Auditor downgrade evaluation** — same; research task, not a v0.1.2 ship item.
- **Model selection / model shootout** — deferred to v0.1.3. Shootout will be license-agnostic (performance ceiling first); user-facing recommendations will then be tiered by license (permissive default / community optional / research-only).
- **PDF support in `source_uri`** — explicitly considered and deferred (see F2 security model). v0.1.2 returns `isError: true` for `application/pdf`; a future version may add optional `pdf` peer dep handling.

## Implementation order

1. **F4 prompt surgery** — smallest change, unblocks immediate usability (classify.reason, summarizer tightness).
2. **F1 `num_ctx`** — config value; single PR.
3. **F3 telemetry footer** — small, enables self-observation on every subsequent call.
4. **F5 savings estimate** — piggybacks on F3 footer + F2 source reader.
5. **F2 file/URL input** — largest; requires `sourceReader.ts` + injection-defense integration + security-limit env vars + tests.

Ship each as a separate PR for clean rollback.

## Dependencies to add

None. All of F1–F5 is accomplished with existing deps (`ollama`, Node built-ins). URL fetch uses Node 22's native `fetch`. No new runtime weight.

## Success criteria

1. **Self-observation round-trip** (primary): in a fresh session after v0.1.2 ships, the frontier (me) delegates ≥ 3 real tasks without being prompted, visibly reads the footer, and course-corrects (e.g. switches to `summarize-long` when footer reports Tier B was near its context cap).
2. **Footer visibility**: every tool call returns a footer line unless `OMCP_TELEMETRY_FOOTER=0`. Unit test checks `content[-1].text` matches `/^\[bridge: /`.
3. **`num_ctx` probe repeatable**: `node tests/probe-numctx.mjs` sends a 7000-word input, asserts `prompt_eval_count >= 7000` (not truncated at 4096). Run per tier.
4. **`classify.reason` fix**: smoke-bridge T4 asserts `reason` is NOT a substring of the input text. Regression test for the echo bug.
5. **`source_uri` end-to-end**: smoke-bridge new T8 — call `summarize-long` with `source_uri: 'file:///tmp/long-doc.txt'`, assert output is non-empty and `_meta[dev.ollamamcpbridge/source_bytes]` matches file size.
6. **Install-size check unchanged**: no new deps; default install still ≤ 50 MB.

## Risks

1. **VRAM OOM on higher `num_ctx`** — qwen2.5:7b at `num_ctx=16384` may push 16 GB Mac past its budget when other apps are running. Mitigation: benchmark during implementation; drop Tier C default to 8 K if measured peak > 12 GB.
2. **`file://` reads bypass client trust boundaries** — some MCP hosts (Claude Desktop sandbox modes) restrict what their MCP servers can do; our reads may be allowed even when the host expected otherwise. Mitigation: clearly document in README that the bridge runs with the user's full filesystem privileges; host sandboxing is the host's job.
3. **URL-fetch as SSRF vector** — if `source_uri: 'http://localhost:8080/admin'` is allowed, malicious prompts could exfiltrate data from the user's local network. Mitigation: env-var denylist for private IP ranges (`OMCP_URL_DENY_PRIVATE=1`, default on); explicit allowlist mode via `OMCP_URL_HOSTS`.
4. **Footer becomes prompt-injection surface** — a hostile source could include text that looks like our footer, tricking the frontier on subsequent calls. Mitigation: footer uses a unique leading sigil (`[bridge:`) unlikely to collide with real text; NFKC normalization on bridge output would be overreach.
5. **`source_uri` + injection defense interaction** — fetched content is inherently untrusted; defender runs on it, but footer's `saved~=+N` estimate could lie if defender blocks (no output returned). Mitigation: emit `saved~=0` when defender blocks.
6. **Self-observation is still N=1** — I'm still the only user. Success criterion #1 could succeed while real users (when they arrive) still struggle. Accept this; v0.1.3 will widen the sample.
7. **CVE-2025-51471 (Ollama, external)** — malicious Ollama registry server can trick the Ollama client into sending its auth token to an attacker-controlled endpoint (discovered 2026-04). The bridge only calls Ollama's chat API and does not expose registry flows, so we are not directly affected. No action in v0.1.2; noted for awareness as an external ecosystem risk.

## Remaining open questions (decide during implementation)

- Footer token cost — 10 vs 12 vs 20 tokens? Measure with tiktoken-equivalent and pick the minimal informative form.
- Should the footer include a one-word quality signal (e.g., `quality=ok|truncated|retry-suggested`) based on heuristics (output short vs input? grammar-constrained but rejected by safeParse?)? Lean no for v0.1.2 — adds complexity.
- Should `source_uri` accept `data:` URIs (base64-inline)? Lean no; MCP already allows inline text via the `text` field.
- Do we want a flag to **force** the footer language to English (for telemetry pipelines) regardless of source language? Lean yes, default English — it's machine-parse territory, not user prose.
