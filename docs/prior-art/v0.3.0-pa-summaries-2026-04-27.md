## PA #1 — Async Job + Persistent Memory Bank

**Use:**
- `p-queue` — bounded concurrency control. Already validated as the right
  shape (we use `p-limit` in v0.2.0 chunking; `p-queue` is its richer sibling
  if we end up needing priority/pause/resume).
- `nanoid` — short, file-system-path-safe job IDs. Better than UUID v4 for
  `.memory/jobs/<id>.md` filenames.
- **Node-native `fs` + `mtime` for 7-day GC** — no DB, no extra layer.
  Boring is correct here.
- **Long-polling pattern (RFC 6202)** for `wait_for_job(max_wait_ms=50000)`.
  Wraps the async-wait in a single MCP tool call; client-facing remains
  request/response.

**Reject:**
- BullMQ — needs Redis. Way out of scope for a single-machine bridge.
- `better-sqlite3` — needs `node-gyp` to compile native bindings; breaks
  the pure-`tsc` build and zero-config cross-platform install promise.
- `node-persist` — adds a file-management abstraction that doesn't earn its
  weight for a 7-day-TTL job log.
- MCP SSE / streamable HTTP at the protocol layer — too complex for what
  long-polling already solves cleanly.

**Skipped middle ground (`ULID`):** time-ordered IDs are nice for log scanning
but not worth the extra dep when nanoid does the job.

## PA #2 — diff-semantic-index + .claude/commands/draft-from-diff

**Use:**
- `parse-diff` — well-trodden npm lib for structuring raw `git diff` output
  into per-file hunks with line markers. Gives us a typed object before we
  feed anything to the local model.
- **Ollama native `format:` JSON Schema** (already in v0.2.0) — re-use the
  same grammar-constrained-output mechanism as `extract` for the
  `diff-semantic-index` tool's structured output. No new mechanism needed;
  this is essentially `extract` with a fixed schema.

**Study (don't depend):**
- `opencommit` chunking strategy for handling diffs larger than the context
  window. Its split heuristic is worth porting.
- `.claude/commands` examples in `alexop/awesome-claude` — `$ARGUMENTS`
  handling and boundary validation patterns for the user-side hook.
- `sourcegraph/cody` repo-context-graph — for v0.5.0+ if we ever want to
  enrich the semantic index with cross-file change topology. Out of scope
  for v0.3.0.

**Reject:**
- `diff2html` — generates HTML, not what we need (we want structured data).
- `CodeBERT` and similar transformer-based code understanding stacks —
  Python + PyTorch, completely off the TS-only path.
- llama.cpp GBNF directly — we already have Ollama's `format:` wrapper;
  don't duplicate.

## Implications for the combined v0.3.0 scope memo

**New runtime deps to add:**
- `p-queue` (small, MIT, already in our dep style)
- `nanoid` (~120 byte runtime, MIT)
- `parse-diff` (small, MIT)

**No changes needed for:**
- Tier B / Tier C model selection
- Existing 6 tools' contracts
- LlmBackend interface (chat / countTokens / ping all suffice)

**Build / install / packaging:**
- All three new deps are pure JS. `npm audit` should stay clean.
- Workspace-root `overrides.uuid` ^14 from v0.2.0 still applies.
- No `node-gyp`, no native bindings, no extra build steps.

**Implementation order recommendation:**
1. `nanoid` + `.memory/jobs/` storage helpers (cheap, foundational)
2. `enqueue-job` MCP tool (calls existing tools internally; just wraps)
3. `wait_for_job` + `read_job_result` (RFC 6202 long-poll on top of #1)
4. `parse-diff` integration + `diff-semantic-index` tool (structured-extract
   with fixed JSON schema)
5. `.claude/commands/draft-from-diff` user-side hook

Roughly tracks the 5-7 d (#1) + 2-3 d (#2) estimates from the v0.3.0
decision doc.
